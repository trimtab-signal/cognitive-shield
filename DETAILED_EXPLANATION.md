## üåê Mesh Network Architecture

### WebRTC Peer-to-Peer

**Technology:** PeerJS (WebRTC signaling)

**How It Works:**
1. Each node generates a unique Peer ID
2. Connects to PeerJS signaling server (free public relay)
3. Peers exchange connection codes (Base64-encoded Peer IDs)
4. Direct WebRTC connection established (no central server)
5. Messages broadcast directly between peers

**Benefits:**
- No central server (resilient)
- Direct connections (low latency)
- Encrypted by default (WebRTC)
- Works across firewalls (STUN/TURN)

**Limitations:**
- Requires STUN server for NAT traversal
- May need TURN server for strict NATs
- Signaling server required (but can be self-hosted)

---

### Dead Man's Switch

**How It Works:**
1. User sets check-in interval (15min, 30min, 1hr, 2hr, 4hr, off)
2. Timer starts after last check-in
3. If timer expires:
   - **1 Miss**: Local alert (notification)
   - **2 Misses**: Broadcast to mesh (peers see "Missed Check-In")
   - **3 Misses**: Trigger escalation webhook (if configured)

**Escalation Options:**
- IFTTT webhook
- Zapier webhook
- Custom API endpoint
- SMS/Email service

**Result:** Trusted peers can detect emergencies and take action.
